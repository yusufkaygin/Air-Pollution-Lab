# Air Pollution Lab

Air Pollution Lab is a client-facing scientific mapping application for exploring air pollution dynamics in Bursa, Turkey over a rolling five-year period.

The project combines a modern React + Leaflet interface with a reproducible Python data pipeline. It is designed for environmental reporting, exploratory spatial analysis, academic case studies, and thesis-oriented interpretation of air quality change.

## What This Project Does

Air Pollution Lab helps users:

- inspect air quality behavior across Bursa on an interactive map
- compare daily, monthly, seasonal, and yearly concentration patterns
- evaluate how pollution changes around specific stations
- study event windows such as wildfire, dust transport, or wind-driven episodes
- relate pollution patterns to roads, green areas, industry, elevation, and local meteorology
- export analytical outputs for reporting and presentation

The interface is intentionally backend-free at runtime. The web application reads a versioned static dataset generated ahead of time by the Python ETL pipeline.

## Product Highlights

- Interactive Bursa-focused map built with Leaflet
- Real data pipeline with static dataset packaging
- Daily, monthly, seasonal, and yearly temporal aggregation
- Station-based contextual analysis with `250 m`, `500 m`, and `1000 m` buffers
- Trend, anomaly, exceedance, and event-window diagnostics
- Context layers for roads, green areas, industry, and elevation
- CSV and PNG export
- Suitable for portfolio demos, stakeholder presentations, and thesis workflows

## Scientific Scope

This project is best used as a spatial-temporal analysis tool for:

- identifying where pollution levels are systematically higher
- comparing periods before, during, and after notable events
- detecting abnormal concentration behavior relative to seasonal history
- assessing whether built environment patterns may be associated with higher concentrations
- supporting thesis sections such as methodology, findings, discussion, and case-study interpretation

For formal scientific claims, the recommended primary source is the official monitoring network. Municipal sensor and modeled layers should be treated as supporting context, not as a replacement for the official archive.

## Data Architecture

The application reads a static dataset from:

- `public/data/bursa-air-quality-v1.json`

That dataset is generated by:

- `scripts/build_static_dataset.py`
- `etl/real_sources.py`
- `etl/pipeline.py`

The pipeline is structured in three layers:

1. raw source acquisition
2. normalization and quality checks
3. static packaging for the frontend

This approach keeps the deployed app lightweight while preserving reproducibility.

## Data Sources

The current pipeline is built around the following sources:

- Official air quality archive: [National Air Quality Monitoring Network](https://sim.csb.gov.tr/STN/STN_Report/StationDataDownloadNew)
- Meteorological archive: [Open-Meteo Archive API](https://archive-api.open-meteo.com/)
- Vector layers: [Overpass API](https://overpass-api.de/)
- Optional fire hotspots: [NASA FIRMS](https://firms.modaps.eosdis.nasa.gov/api/)

The project may also include curated event windows derived from verified public sources when a complete machine-readable archive is not available.

## Key Analytical Methods

Air Pollution Lab includes:

- concentration aggregation by day, month, season, and year
- station climatology-based anomaly scoring
- weekday versus weekend comparison
- exceedance-day and exceedance-episode detection
- Mann-Kendall style trend interpretation
- Seasonal Kendall trend summary
- CUSUM-style structural break detection
- KZ decomposition for background vs short-term variability
- contextual correlation summaries against station buffer metrics
- event-window comparison around selected incidents

These methods are intended for exploratory and decision-support analysis. They support scientific interpretation, but results should still be discussed with methodological caution in a thesis or report.

## Getting Started

Install dependencies and run the project locally:

```powershell
python -m pip install -r requirements.txt
npm install
npm run generate:data
npm run dev
```

Production build:

```powershell
npm run build
npm run preview
```

## Available Scripts

- `npm run dev` starts the local development server
- `npm run build` builds the production bundle
- `npm run preview` previews the production build locally
- `npm run lint` runs ESLint
- `npm run test` runs frontend tests
- `npm run test:py` runs Python pipeline tests
- `npm run generate:data` fetches and packages the current Bursa dataset

## Data Generation

Default dataset build:

```powershell
python scripts/build_static_dataset.py `
  --mode fetch `
  --output public/data/bursa-air-quality-v1.json
```

Build a specific time window manually:

```powershell
python scripts/build_static_dataset.py `
  --mode fetch `
  --start-date 2021-03-15 `
  --end-date 2026-03-15 `
  --output public/data/bursa-air-quality-v1.json
```

Local import mode is also available for manually curated files:

```powershell
python scripts/build_static_dataset.py `
  --mode local `
  --output public/data/bursa-air-quality-v1.json `
  --air-quality-csv data/raw/air_quality.csv `
  --meteo-csv data/raw/meteo.csv `
  --context-csv data/raw/context_metrics.csv `
  --events-csv data/raw/events.csv
```

## Recommended Use in Research

This project can support a thesis or scientific report in several ways:

- as a visual exploration tool for identifying candidate case studies
- as a comparison environment for pre-event and post-event behavior
- as a supporting system for interpreting spatial heterogeneity across districts
- as a dashboard for explaining relationships between pollution and urban context
- as a structured export source for charts, tables, and summaries

Good thesis use cases include:

- seasonal PM10 or PM2.5 comparisons
- station-based anomaly detection
- wildfire or dust-event case windows
- built-environment interpretation using roads, green areas, and industry
- trend discussion across the last five years

## Project Structure

- `src/App.tsx`: application shell and dashboard orchestration
- `src/components/MapPanel.tsx`: Leaflet map, spatial layers, and map UI
- `src/components/InsightsPanel.tsx`: charts, analytical summaries, and tables
- `src/utils/analytics.ts`: aggregation, diagnostics, trend logic, and event analysis
- `src/utils/mapPlumes.ts`: organic map plume rendering for pollution visualization
- `etl/real_sources.py`: live-source fetching and static packaging
- `etl/pipeline.py`: local import normalization pipeline
- `scripts/build_static_dataset.py`: dataset build entry point

## Deployment

Because the app is static at runtime, it can be deployed easily on services such as:

- Vercel
- GitHub Pages

Vercel is the preferred option for this project because it handles the current dataset size more comfortably.

## Notes on Reliability

- The frontend does not scrape live data during runtime.
- The application reads a generated static snapshot.
- Official monitoring data should remain the primary basis for scientific interpretation.
- Missing records are not silently imputed; data completeness is surfaced in the interface.
